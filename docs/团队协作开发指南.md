## 1. 问题分析与解决方案总览

### 1.1 问题一：如何协作开发，避免代码冲突和接口失效

**问题描述：**
- 直接在 GitHub 上 commit 可能导致代码冲突
- 模型改动可能导致 API 接口失效
- 前端改动时无法预知是否会报错

**解决方案：**

| 方案 | 具体措施 | 优先级 |
|------|----------|--------|
| **分支管理** | 采用 Git Flow 工作流，每人一个功能分支 | 高 |
| **接口契约** | 定义 API 接口规范，改动需先评审 | 高 |
| **类型注解** | 添加 Python 类型注解，IDE 可提前发现错误 | 中 |
| **单元测试** | 关键接口添加测试，PR 自动运行 | 中 |
| **变更通知** | 接口改动需在群内通知 + 文档更新 | 高 |

### 1.2 问题二：分时间段开发是否可行

**问题描述：**
- 想过分时间段，一个时间段一个人完成
- 时间紧张，可能来不及预期内实现功能

**解决方案：**

| 方案 | 具体措施 | 时间估算 |
|------|----------|----------|
| **并行开发** | 按模块划分，多人同时开发不同模块 | 节省 60% 时间 |
| **接口先行** | 先定义接口，再并行实现 | 减少返工 |
| **每日站会** | 15 分钟同步进度，及时发现问题 | 避免延期 |
| **里程碑管理** | 设置周里程碑，跟踪完成情况 | 保证进度 |

### 1.3 问题三：代码结构臃肿，缺乏工程性

**问题描述：**
- src 部分文件夹过多，文件相互包含
- 结构不够清晰，缺乏鲁棒性和泛化能力
- 是赶工出来的结构，现在难以大改

**解决方案：**

| 方案 | 具体措施 | 工作量 |
|------|----------|--------|
| **文档化结构** | 绘制模块依赖图，明确各模块职责 | 低 |
| **渐进式重构** | 不推翻重来，逐步优化关键路径 | 中 |
| **统一入口** | 定义清晰的模块入口，减少交叉引用 | 中 |
| **添加类型注解** | 提升代码可读性和 IDE 支持 | 低 |
| **编写模块文档** | 每个模块添加 docstring 说明 | 低 |

## 2. Git 协作开发流程

### 2.1 推荐工作流：GitHub Flow 简化版

```
main 分支（受保护）
  │
  ├─── feature/xxx-功能名 (开发者 1)
  ├─── feature/yyy-功能名 (开发者 2)
  ├─── feature/zzz-功能名 (开发者 3)
  │
  └─── 每次合并需经过 Pull Request + Code Review
```

### 2.2 分支命名规范

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| 功能分支 | `feature/模块名 - 功能描述` | `feature/models-add-gru` |
| 修复分支 | `fix/问题描述` | `fix/api-predict-error` |
| 文档分支 | `docs/文档内容` | `docs/api-specification` |
| 重构分支 | `refactor/重构内容` | `refactor/src-structure` |

### 2.3 开发流程

```bash
# 1. 从 main 拉取最新代码
git checkout main
git pull origin main

# 2. 创建功能分支
git checkout -b feature/your-feature-name

# 3. 开发并提交（建议小步提交）
git add .
git commit -m "feat: 添加 XX 功能"

# 4. 推送分支
git push origin feature/your-feature-name

# 5. 在 GitHub 上创建 Pull Request
# 6. 等待 Code Review 后合并
```

### 2.4 Commit Message 规范

采用 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**Type 类型：**

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `style` | 代码格式（不影响功能） |
| `refactor` | 重构 |
| `test` | 测试相关 |
| `chore` | 构建/工具/配置 |

**示例：**

```bash
git commit -m "feat(models): 添加 GRU 模型实现"
git commit -m "fix(api): 修复预测接口返回值错误"
git commit -m "docs(readme): 更新安装说明"
git commit -m "refactor(services): 重构 ForecastService 减少耦合"
```

### 2.5 避免冲突的最佳实践

**1. 拉取最新代码再开发**
```bash
# 每天开始工作前
git checkout main
git pull origin main
git checkout your-branch
git rebase main  # 变基到最新
```

**2. 小步提交，频繁同步**
```bash
# 每完成一个小功能就提交
git add <specific-file>
git commit -m "feat: 完成 XX 小功能"

# 每天至少同步一次 main 分支
```

**3. 避免修改同一文件**
- 团队内提前沟通谁修改哪个文件
- 使用 `git blame` 查看文件最后修改者

#### 3.3.1 模型组接口规范

```python
# src/models/base_model.py
from abc import ABC, abstractmethod
from typing import Dict, Any
import numpy as np

class BaseForecastModel(ABC):
    """所有模型必须实现的基类"""
    
    @property
    @abstractmethod
    def name(self) -> str:
        """模型名称"""
        pass
    
    @abstractmethod
    def train(self, X_train: np.ndarray, y_train: np.ndarray, 
              **kwargs) -> Dict[str, Any]:
        """训练模型"""
        pass
    
    @abstractmethod
    def predict(self, X: np.ndarray, **kwargs) -> np.ndarray:
        """预测"""
        pass
    
    @abstractmethod
    def save(self, path: str) -> None:
        """保存模型"""
        pass
    
    @abstractmethod
    def load(self, path: str) -> None:
        """加载模型"""
        pass
```

## 4. 代码结构重构计划

### 4.1 当前问题分析

**问题 1：目录结构冗余**
```
# 问题示例：相似功能分散在多处
src/data/feature_engineering.py       # 特征工程 1

```

**问题 2：循环依赖**
```
# 可能的循环依赖
src/services/forecast_service.py → src/models/model_service.py
src/models/model_service.py → src/services/forecast_service.py
```

**问题 3：模块职责不清**
```
# model_layer 和 models 职责重叠
# data 和 data_layer 职责重叠
```

### 4.2 渐进式重构方案

**原则：不推翻重来，逐步优化**

#### 第一阶段：文档化（1-2 天）

1. **绘制模块依赖图**
```
┌─────────────┐
│   app.py    │  前端入口
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  services/  │  服务层（核心编排）
└──────┬──────┘
       │
       ├───► ┌─────────────┐
       │     │   models/   │  模型层
       │     └─────────────┘
       │
       ├───► ┌─────────────┐
       │     │ data_layer/ │  数据层
       │     └─────────────┘
       │
       └───► ┌─────────────┐
             │  logic/     │  交易逻辑
             └─────────────┘
```

2. **添加模块文档**
```python
# src/services/__init__.py
"""
服务层模块

职责：
- 统一编排模型层、数据层、交易逻辑层
- 提供高层业务接口

依赖：
- models/*: 模型实现
- data_layer/*: 数据处理
- logic/*: 交易逻辑

被依赖：
- app_layer/*: API 服务
- app.py: 前端
"""
```

#### 第二阶段：统一入口（2-3 天）

1. **为每个模块添加 `__init__.py` 导出**

```python
# src/models/__init__.py

    "LSTMForecastModel",
    "GRUForecastModel",
    "XGBoostForecastModel",
    "MoiraiForecastModel",
    "StackingManager",
]
```

2. **外部引用改为从入口导入**

```python
# 改动前
from src.models.lstm_model import LSTMForecastModel
from src.models.stacking_manager import StackingManager

# 改动后
from src.models import LSTMForecastModel, StackingManager
```

#### 第三阶段：清理冗余（3-5 天）

## 附录 B：常见问题解答

**Q1: 合并冲突了怎么办？**

A: 先拉取最新代码，本地解决冲突后再提交：
```bash
git pull origin main
# 解决冲突文件中的 <<<<<<< 和 >>>>>>> 标记
git add <resolved-files>
git commit -m "fix: 解决合并冲突"
git push
```

**Q2: 不小心提交了敏感信息（如密码）怎么办？**

A: 
1. 立即修改密码
2. 使用 `git filter-branch` 或 BFG 清理历史记录
3. 强制推送到远程

**Q3: 如何回退错误的提交？**

A:
```bash
# 回退最后一次提交，保留更改
git reset --soft HEAD~1

# 回退并丢弃更改
git reset --hard HEAD~1

# 已经推送？创建反向提交
git revert <commit-hash>
git push
```
